# QBB 工作报告（2026-07-21 17:09）

## 一、本轮工作的目标

本轮工作将 C 侧从“Portfolio Manager 之后再增加一层硬编码决策闸门”的架构，
调整为“由多智能体保留最终决策权，同时通过工具证据、结构化主张和确定性审计提高
报告可信度”的架构。

Portfolio Manager 仍然是最终 Rating 的唯一权威。新增可信度模块只负责记录、
核验和披露问题，不会修改 `Buy / Overweight / Hold / Underweight / Sell`。

## 二、完成的主要内容

### 1. 清理旧 BHD 硬编码决策层

旧 Hybrid/BHD 决策闸门及其运行路径已经从默认工作流、CLI、配置和报告中移除。
原本有实际分析价值的行情快照、量价特征和情绪词典能力被迁移到 Analyst Tools，
供 Agent 按职责使用。

公共 `DecisionProvider` 协议仍然保留，方便 A/B 后续接入 Broker、模拟盘或薄策略
层；当前默认图不再提供或调用 HybridDecisionProvider。

### 2. 扩展 Analyst 数据与工具

- Market Analyst 可以使用多周期 OHLCV、量能异常、价格缺口和校验快照工具。
- Sentiment Analyst 强制预取新闻，并计算可复现的词典情绪先验。
- News Analyst 强制预取公司与全球新闻；Google News 会组合 ticker、公司别名、
  业绩、指引、监管和产品等查询并去重。
- Fundamentals Analyst 使用 SEC 数据路径获取公司资料和主要财务报表。
- Bull/Bear 仅保留基准比较和回撤风险两个非重叠工具；每个工具在一次完整 run
  中最多执行一次。

默认数据路径已调整为无需行情 Key 的 `yahoo_chart`、`google_news` 和
`sec_edgar`。代码不应再默认假设 `yfinance` 是当前启用的数据源。

### 3. 增加确定性统计附录和时序图

保存报告时会生成：

- `appendix/statistics.md`
- `appendix/price_volume.png`

总报告第 VI 节包含统计表和图片相对链接。CLI 保存完成后也会显示图片的实际路径。
附录如果在报告保存阶段重新取数，会被记录为 `post_run_recomputed`，不会反向作为
运行时正文的证据。

### 4. 实现多智能体可信度 P0–P3

可信度实现位于：

`tradingagents/extensions/decision/credibility/`

主要能力包括：

- 每次运行生成唯一 `run_id`。
- ToolNode、Sentiment/News 预取、Bull/Bear 自定义工具循环和报告附录统一生成
  `TraceEvent`。
- 工具参数会脱敏，结果会保存为带内容 hash 的 Artifact。
- structured output 会记录成功、自由文本回退、错误类型和尝试次数。
- Research Manager、Trader、Portfolio Manager 会生成 `DecisionSnapshot`。
- 严格 Rating 解析失败时不会静默默认成 Hold。
- Analyst 会生成 Typed Claims、字段级 `ArtifactFieldRef` 和不确定性 sidecar。
- Numeric、Temporal、Source、Structured 四类规则由统一 Verifier 执行。
- AuditProfile 分维度披露审计结果，不生成单一“可信度总分”。
- Bull/Bear 会记录 `DebateTurnRecord`；完整 Bull+Bear 周期没有新增证据时可以
  提前结束。
- Research Manager Schema 支持 accepted/rejected/unresolved claim IDs。
- Risk Agent 生成 constraint ID，供 Portfolio Manager 解释决策变化。

### 5. 完整审计输出

每次保存报告会新增 `audit/` 目录：

- `event_trace.jsonl`：规范事件流。
- `artifacts/`：工具结果原文。
- `claims.jsonl`：结构化主张。
- `findings.jsonl`：确定性核验结果。
- `profile.json`：AuditProfile 和指标分母。
- `manifest.json`：文件清单、数量和 hash。
- `credibility.md`：供读者阅读的审计摘要。

总报告第 VII 节会嵌入可信度摘要。`AUDIT_FAILED` 表示存在 Critical Finding，
不表示 PM Rating 被自动判错或覆盖。

## 三、验证结果

- 与本轮改动直接相关的回归测试：`91 passed`。
- 全仓非网络测试：`571 passed`。
- AAPL 与 TSLA 均完成真实 DeepSeek Flash CLI 运行并生成完整报告。
- 两轮报告均包含统计附录、价量图、审计目录、Claims、Findings、AuditProfile
  和 Bull/Bear 工具使用记录。
- TSLA 报告的关键主张证据覆盖率达到 81.9%，审计器识别出字段级数值引用不一致，
  说明审计链路能够实际暴露报告问题。
- PM 原始 `**Rating**: Hold` 在保存和审计过程中保持不变。

全仓剩余测试问题集中在既有 dataflow 默认 vendor 与旧 yfinance 假设之间的兼容，
不属于可信度规则对 PM 决策的修改。

## 四、A/B 开发者需要注意的接口与行为变化

### 1. AgentState 增加了审计字段

新增字段包括：

- `run_id`
- `audit_events`
- `structured_invocations`
- `decision_snapshots`
- `claims`
- `audit_scope_reasons`

请优先通过 `Propagator.create_initial_state()` 创建运行状态。自行构造 state 的代码
应允许这些新增字段存在，并避免按固定字段集合覆盖整个 state。

`InvestDebateState` 还增加了工具预算、DebateTurnRecord 和 Claim 处理状态；
`RiskDebateState` 增加了 `constraints`。

### 2. ToolNode 外部增加了审计包装

Graph 中的 Analyst ToolNode 现在由审计函数包装，但仍保留 `tools_by_name` 兼容
入口。集成代码应将节点视为可调用 Runnable，而不要依赖其必须是原始
`langgraph.prebuilt.ToolNode` 实例。

### 3. Structured helper 保持旧接口兼容

原有：

`invoke_structured_or_freetext(...) -> str`

仍然保留。新增：

`invoke_structured_with_metadata(...) -> StructuredInvocationResult`

需要审计元数据的新节点使用后者；只需要文本的旧调用方无需修改。

### 4. 报告保存现在会产生更多文件

`write_report_tree()` 和 `TradingAgentsGraph.save_reports()` 会额外写入
`appendix/`、`audit/`，并在保存后的 `final_state` 中加入 `audit_profile`。

A 的模拟盘或 Broker 集成如果监听报告目录，需要忽略未知子目录，或明确消费
`audit/profile.json`。B 的记忆模块如果存储最终报告，应区分 PM 原始决策和审计
投影，不要把 Findings 改写成新的交易决策。

### 5. PM Rating 的权限边界不变

以下字段仍是交易结论来源：

- `risk_debate_state.judge_decision`
- `final_trade_decision`

AuditProfile、Finding、主观 Confidence、词典分数和量化特征都不是第二个交易
决策。A/B 集成不得用 `AUDIT_FAILED` 自动替换 PM Rating；可以据此暂停执行、
要求人工复核或降低自动化等级，但应保留原始决策和原因。

### 6. 数据源默认值发生变化

当前默认路径优先使用：

- 行情：`yahoo_chart`
- 新闻：`google_news`
- 美股基本面：`sec_edgar`

测试或集成代码如需指定其他 vendor，应显式设置配置，不应依赖历史默认值。

### 7. 报告章节编号发生变化

- 第 V 节：Portfolio Manager Decision
- 第 VI 节：确定性统计与价量图
- 第 VII 节：可信度审计

解析报告的代码应优先读取结构化 state 或审计 JSON，不应只依靠 Markdown 标题
位置或固定章节数量。

## 五、协作建议

- A 在接入模拟交易前，可以读取 `audit/profile.json` 和 Critical Findings 决定
  是否进入人工确认流程，但不要重写 PM Rating。
- B 在存储长期记忆时，应保留 `run_id`、DecisionSnapshot、PM 原文和审计状态，
  便于后续回放同一次运行。
- 合并其他分支时重点检查 `agent_states.py`、`propagation.py`、
  `trading_graph.py`、`reporting.py` 和数据 vendor 配置，这些是跨团队最容易发生
  冲突的文件。
- 新工具必须接入统一 Evidence Observation；新增决策节点应生成
  StructuredInvocationResult 和 DecisionSnapshot。
- 可信度规则只验证其声明的范围。不要将来源一致、字段一致或时间安全解释成对
  金融因果、未来收益或事实真伪的全面证明。
