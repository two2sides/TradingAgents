# TradingAgents 主体代码文档

本文档覆盖仓库**主体业务代码**（`tradingagents/` 包与根目录 `main.py`），**不包含** CLI/UI（`cli/`）、测试（`tests/`）及辅助脚本。描述以自然语言说明每个文件「负责什么、实现了什么」，便于阅读与二次开发。

---

## 一、总览

TradingAgents 是一套基于 **LangGraph** 的多智能体投研框架。一次典型运行会：

1. 用配置与环境变量初始化「深度思考 / 快速思考」两套 LLM；
2. 拉取并校验行情、新闻、基本面等数据（Agent 通过 Tool 调用）；
3. 按流水线依次执行：分析师团队 → 多空辩论 → 研究经理 → 交易员 → 风控辩论 → 组合经理；
4. 解析五档评级，写入记忆日志与报告树；后续运行可对 pending 决策做收益反思。

决策主体是 LLM；数据层与部分校验（指标、快照、符号规范化）是确定性计算。

```
main.py
  └─ TradingAgentsGraph.propagate(ticker, date)
       ├─ llm_clients/          创建 LLM
       ├─ dataflows/ + agents/utils/*_tools.py   数据 Tool
       ├─ graph/setup.py        拼装 LangGraph
       ├─ agents/*              各角色节点
       ├─ agents/utils/memory.py  决策记忆
       └─ reporting.py          落盘报告
```

## 二、入口与包根

### `main.py`

最简编程入口：复制默认配置，构造 `TradingAgentsGraph`，对示例标的（如 NVDA）与交易日调用 `propagate`，打印最终评级字符串。用于无 CLI 的快速试跑。

### `tradingagents/__init__.py`

包导入时加载项目目录下的 `.env` / `.env.enterprise`（不覆盖已有环境变量），并压制部分 langchain/langgraph 启动时的弃用告警，保证无论从 `main.py` 还是安装后的入口启动都能读到 API Key。

### `tradingagents/default_config.py`

全局默认配置字典 `DEFAULT_CONFIG`：结果目录、缓存目录、记忆日志路径、LLM 提供商与模型名、辩论轮数、新闻条数、数据 vendor 路由、基准指数映射等。通过 `TRADINGAGENTS_*` 环境变量按类型自动覆盖对应键，是无代码改配置的主通道。

### `tradingagents/reporting.py`

把一轮运行结束后的 `final_state` 写成目录树：分析师、研究、交易、风控、组合各小节 markdown，并汇总 `complete_report.md`。CLI 与图 API 共用，保证无头运行也能得到与交互运行一致的报告结构。

## 三、图编排层 `tradingagents/graph/`

### `trading_graph.py`

核心编排类 `TradingAgentsGraph`：

- 初始化双 LLM、工具节点、条件路由、图编译、记忆日志、反射器、信号处理器；
- `propagate`：解析标的身份、注入历史记忆、可选 checkpoint 断点续跑、执行图、落报告、写入 pending 决策；
- 新一轮同标的运行时，尝试用行情结算历史 pending 条目并生成反思（Phase B）；
- 按交易所后缀选择 alpha 基准（如美股 SPY、港股恒指等）。

这是「程序怎么跑起来」的总控。

### `setup.py`

用 LangGraph `StateGraph` 把各 Agent 与 Tool 节点连成有向图：分析师按选择顺序串行（每人「思考 ↔ 调工具」循环），再接多空研究员、研究经理、交易员、三方风控、组合经理。辩论边使用完整 path map，避免路由标签漂移导致运行中崩溃。

### `conditional_logic.py`

图上的条件边：某分析师最后一条消息若含 tool_calls 则进对应 tools 节点，否则清消息并进入下一阶段；多空辩论与风控辩论根据轮次与「下一位发言者」标签决定继续辩论还是交给经理/组合经理。

### `propagation.py`

构造图的初始 `AgentState`（公司、日期、资产类型、instrument 上下文、空报告字段、空辩论状态等），并配置 LangGraph 调用参数（如递归上限）。

### `analyst_execution.py`

根据用户勾选的分析师列表生成执行计划（节点名、清消息节点名、工具节点名顺序）；另有墙钟耗时跟踪器，供 CLI 展示各分析师耗时（主体图逻辑也依赖执行计划构建边）。

### `checkpointer.py`

基于 SQLite 的 LangGraph checkpoint：按标的存库、按 ticker+date+图形状签名生成 thread id，支持崩溃后续跑；图形状（分析师集合、辩论深度等）变化会换签名，避免错误续跑。

### `reflection.py`

`Reflector`：在已知真实收益与相对基准 alpha 后，用短提示让 LLM 写 2–4 句反思，供写入记忆日志并在未来 run 中作为 past context。

### `signal_processing.py`

从组合经理最终长文中**确定性**解析五档评级（Buy / Overweight / Hold / Underweight / Sell），不再二次调 LLM；兼容旧的 `process_signal` 接口。

### `graph/__init__.py`

对外导出图相关主类（`TradingAgentsGraph`、`GraphSetup`、`Propagator` 等）。

## 四、智能体层 `tradingagents/agents/`

### `agents/__init__.py`

汇总导出各 `create_*` 工厂与状态类型，供 `setup.py` 引用。

### `agents/schemas.py`

结构化输出的 Pydantic 模型：研究计划、交易提案、组合决策、情绪报告及渲染回 markdown 的辅助函数。保证关键决策角色输出带稳定表头（尤其是 Rating），便于下游解析与展示。

### 分析师 `agents/analysts/`

| 文件 | 作用 |
|------|------|
| `market_analyst.py` | 技术面分析师：引导模型选指标、先拉行情再算指标，并要求做已校验市场快照，产出 `market_report`。 |
| `fundamentals_analyst.py` | 基本面分析师：调用财报、现金流量表、利润表、概况等工具，产出 `fundamentals_report`。 |
| `news_analyst.py` | 新闻/宏观分析师：公司新闻、全球宏观新闻、内部人交易、FRED 宏观、预测市场等，产出 `news_report`。 |
| `sentiment_analyst.py` | 情绪分析师：聚合新闻与社媒类信号，结构化情绪带与报告（可渲染 markdown）。 |
| `social_media_analyst.py` | 旧名兼容层，转发到 `sentiment_analyst`。 |

每个文件模式类似：`create_xxx(llm)` 返回一个读 `state`、绑 tools、调 LLM、写回报告字段的节点函数。

### 研究员 `agents/researchers/`

| 文件 | 作用 |
|------|------|
| `bull_researcher.py` | 多头：基于四份分析师报告与辩论历史，强化上涨论据，更新 `investment_debate_state`。 |
| `bear_researcher.py` | 空头：对称地强化风险与下跌论据。 |

### 经理 `agents/managers/`

| 文件 | 作用 |
|------|------|
| `research_manager.py` | 裁决多空辩论，产出结构化投资计划 `investment_plan`（含推荐档位与行动建议）。 |
| `portfolio_manager.py` | 综合风控三方辩论与交易计划，产出最终 `final_trade_decision`（含五档 Rating）。 |

### 交易员 `agents/trader/trader.py`

把研究经理计划转成可执行交易提案（Buy/Hold/Sell 及仓位思路），写入 `trader_investment_plan`。

### 风控 `agents/risk_mgmt/`

| 文件 | 作用 |
|------|------|
| `aggressive_debator.py` | 激进风险观点：更倾向抓住上行、接受波动。 |
| `conservative_debator.py` | 保守风险观点：强调下行保护与仓位克制。 |
| `neutral_debator.py` | 中性风险观点：平衡双方，推动可执行折中。 |

三人轮流更新 `risk_debate_state`，最后由组合经理收口。

## 五、Agent 工具与状态 `tradingagents/agents/utils/`

### `agent_states.py`

定义图状态：`AgentState`（消息、标的、日期、四份报告、投资/风控辩论子状态、交易计划、最终决策、past_context 等）、`InvestDebateState`、`RiskDebateState`。

### `agent_utils.py`

横切工具：输出语言指令、解析 yfinance 标的身份、构造 instrument 上下文字符串、从 state 读取上下文、以及「清空消息历史」节点工厂（分析师阶段结束后防止 tool 对话污染下一节点）。

### `memory.py`

`TradingMemoryLog`：追加式 markdown 决策日志。运行结束写 pending 决策；之后用真实收益更新并附加 REFLECTION；`get_past_context` 取同标的完整条目 + 跨标的教训摘要，注入下一轮初始 state。支持条目上限轮转与原子写，避免损坏日志。

### `rating.py`

五档评级词表与从自由文本中启发式解析 Rating 的函数（供信号处理器与记忆标签共用）。

### `structured.py`

结构化输出封装：尽量用 provider 原生 structured 模式；失败则回退自由文本并尽力解析，避免整条流水线因 schema 失败中断。

### 各类 `*_tools.py`（LangChain `@tool` 薄封装）

| 文件 | 暴露的工具 |
|------|------------|
| `core_stock_tools.py` | `get_stock_data` OHLCV |
| `technical_indicators_tools.py` | `get_indicators` 技术指标 |
| `fundamental_data_tools.py` | 基本面与三大报表 |
| `news_data_tools.py` | 公司新闻、全球新闻、内部人交易 |
| `macro_data_tools.py` | 宏观指标 |
| `prediction_markets_tools.py` | 预测市场 |
| `market_data_validation_tools.py` | 已校验市场快照（防幻觉引用） |

这些工具内部统一 `route_to_vendor`，不直接绑死某一数据商。

## 六、数据流层 `tradingagents/dataflows/`

### `interface.py`

数据访问契约中枢：按工具类别与配置中的 vendor 链路由到具体实现（yfinance、Alpha Vantage、FRED、Polymarket 等），处理限流与未配置错误，避免静默换源造成结果不可复现。

### `config.py`

进程内持有一份可 `set_config` / `get_config` 的配置副本，供 dataflows 在 Tool 调用时读缓存路径、vendor 等。

### `errors.py`

数据错误分类：无行情、限流、vendor 未配置等，便于上层区分「可重试」与「配置问题」。

### `utils.py`

日期辅助、DataFrame 保存、以及把 ticker 变成安全路径组件（防路径穿越）等杂项。

### `symbol_utils.py`

把用户/券商风格符号规范到 Yahoo 等数据源可识别形式（加密货币、外汇别名、CFD 尾缀等），并提供「是否适合直接打 Yahoo」的判断。

### `stockstats_utils.py`

拉取并清洗 OHLCV、用 stockstats 算指标、按 `curr_date` 截断以防回测前瞻偏差、检测陈旧行情帧、金融表按日过滤。是技术分析数值层的核心。

### `market_data_validator.py`

基于真实 OHLCV+指标生成「已核验市场快照」文本，约束分析师引用价格/指标时有据可查。

### `y_finance.py`

yfinance 实现：在线行情、指标窗口、基本面概况、三大报表、内部人交易等，并附带各指标用途说明字符串（供分析师提示词引用）。

### `yfinance_news.py`

基于 yfinance 的公司新闻与全球宏观新闻拉取，带时间窗过滤，降低历史回放时的前瞻泄露。

### Alpha Vantage 系列

| 文件 | 作用 |
|------|------|
| `alpha_vantage.py` | 聚合导出 AV 侧各能力。 |
| `alpha_vantage_common.py` | API Key、请求、限流、CSV 按日期过滤。 |
| `alpha_vantage_stock.py` | 股票行情。 |
| `alpha_vantage_indicator.py` | 技术指标。 |
| `alpha_vantage_fundamentals.py` | 基本面与报表（含日期过滤）。 |
| `alpha_vantage_news.py` | 新闻、全球新闻、内部人交易。 |

### 其它 vendor

| 文件 | 作用 |
|------|------|
| `fred.py` | 美联储 FRED 宏观序列。 |
| `polymarket.py` | Polymarket 预测市场检索（偏前瞻事件）。 |
| `reddit.py` | 按标的抓取 Reddit 讨论（RSS/JSON，带重试）。 |
| `stocktwits.py` | StockTwits 公开消息流，供情绪分析。 |

## 七、LLM 客户端层 `tradingagents/llm_clients/`

### `__init__.py` / `factory.py`

按 provider 字符串创建对应客户端；工厂延迟导入重依赖，避免 import 阶段过重。

### `base_client.py`

抽象基类：统一 `get_llm()` 与内容规范化，屏蔽各 SDK 返回形态差异。

### 各厂商客户端

| 文件 | 作用 |
|------|------|
| `openai_client.py` | OpenAI 及大量 OpenAI 兼容端点（DeepSeek、Qwen、GLM、本地 Ollama 等），含 reasoning effort、内容规范化。 |
| `anthropic_client.py` | Claude，含 effort 参数版本门控。 |
| `google_client.py` | Gemini，含 thinking level。 |
| `azure_client.py` | Azure OpenAI。 |
| `bedrock_client.py` | Amazon Bedrock（可选依赖 langchain-aws）。 |

### 支撑模块

| 文件 | 作用 |
|------|------|
| `api_key_env.py` | provider → 环境变量名映射与探测。 |
| `model_catalog.py` | CLI/校验用的模型目录。 |
| `validators.py` | 模型名是否在已知列表。 |
| `capabilities.py` | 声明式能力表（是否支持某参数等），驱动 OpenAI 兼容路径差异化行为。 |

## 八、阅读建议

1. 先读 `default_config.py` 与 `graph/trading_graph.py`，建立「配置 → 跑图 → 记忆/报告」心智模型。  
2. 再读 `graph/setup.py` + `agents/utils/agent_states.py`，理解节点顺序与状态字段。  
3. 数据侧从 `dataflows/interface.py` 下钻到具体 vendor。  
4. 记忆与反思看 `agents/utils/memory.py` + `graph/reflection.py`。  

与本仓库配套的**一次完整执行模拟**见同目录 `执行流程图.md`；C 侧扩展见
`扩展方案.md`，可信度架构见 `多智能体可信度设计方案.md`。
