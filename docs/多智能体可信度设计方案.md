# TradingAgents 多智能体可信度设计方案

> 适用范围：当前 TradingAgents 工作流与 C 侧工具、报告增强。  
> 设计目标：让系统的每个重要结论都能回答“依据是什么、何时取得、由谁产生、能否复核、冲突在哪里”。  
> 本文是设计与验收规范，不代表所有组件已经实现。文中明确标注“已有 / 待开发”。

---

## 1. 为什么需要专门的可信度层

当前系统已经具备完整的多智能体链路：

```text
Market / Sentiment / News / Fundamentals
    → Bull / Bear
    → Research Manager
    → Trader
    → Risk 三人
    → Portfolio Manager
    → 报告与记忆
```

这个结构能够产生内容丰富的研究报告，但“参与 Agent 更多”并不自动意味着“更可信”。近期多智能体研究反复指出：

1. 多个 Agent 可能使用同一个底层模型、相似训练数据和相似提示词，因此错误高度相关；
2. 辩论可能因从众、讨好或追求获胜而放大错误，而不是纠正错误；
3. 最终文字很有说服力，不代表它真实反映了工具证据；
4. 只检查最终 Rating，无法解释错误最早出现在数据、分析、辩论还是交接阶段；
5. LLM 自报“置信度 90%”没有经过校准，不能直接当作可信度。

因此本方案不增加“另一个更聪明的裁判 LLM”，而是建立一套**证据来源、主张核验、流程一致性、不确定性披露和压力测试**机制。

可信度层只回答：

- 数据和主张是否有来源；
- 来源是否满足分析日期；
- 数字是否与工具结果一致；
- 上下游决策是否发生无解释的方向变化；
- 工具或结构化输出是否失败；
- 哪些分歧仍未解决；
- 这份报告是否达到“可验证”标准。

可信度层**不预测涨跌，不覆盖 Portfolio Manager 的 Rating，也不恢复已经删除的 BHD/量化闸门**。

## 2. 研究依据与可借鉴边界

### 2.1 多智能体辩论并非天然可靠

**Multi-LLM Debate: Framework, Principals, and Interventions（NeurIPS 2024）**
在其理论模型和四个通用基准上表明，相似能力、相似回答及共享误解可能使辩论
停滞或收敛到错误多数，并研究了 diversity pruning、quality pruning 和
misconception refutation 三类干预。

对本项目的启示：

- Bull/Bear 的角色相反不等于证据独立；
- 不能用“多数 Agent 都同意”作为真实性证明；
- 应衡量双方是否提出了不同且有效的证据，而不是只统计立场。

限制：

- 论文主要研究通用推理任务，不能证明其干预会提高金融预测或交易收益；
- 当前项目主要使用同一供应商/模型，因此只能先增加**证据多样性**，不能声称实现了模型异质性。

### 2.2 辩论可能导致从众和错误收敛

**Talk Isn’t Always Cheap: Understanding Failure Modes in Multi-Agent Debate**
（ICML 2025 Multi-Agent Systems workshop）在若干通用推理任务上观察到，
部分异构模型辩论会随轮次降低准确率，并出现正确答案转为错误答案的现象。
社会影响、顺从式更新、模型能力和任务类型与该现象相关；论文没有证明谄媚是
唯一或确定的原因。

**When and Why Does Multi-Agent Debate Fail and Does It Really Underperform?**
（arXiv 预印本，初版 2025；初步工作发表于 ICML 2026 workshop）进一步区分：

- 竞争式辩论可能退化为“为了赢而说服”；
- 共识式辩论可能过早抹掉有价值的分歧；
- 协作式、非零和协议更有利于提供真实信息。

对本项目的启示：

- Bull/Bear 不应以“赢得辩论”为目标；
- Research Manager 必须保留未解决分歧，不能把结论写成虚假共识；
- 后续轮次必须增加新证据或指出具体错误，否则应停止重复争论。

这些结论来自特定博弈假设和错误检测、数学推理、安全任务实验，不能直接证明
投资多空辩论采用协作提示后就会得到更准确的市场判断。

### 2.3 事实核验需要主张级证据，而不是再写一份长文

**Debating Truth / DebateCV（WWW 2026）**采用支持方、反对方和经过
Debate-SFT 训练的 Moderator，对给定证据集合上的复杂事实主张进行裁决。

**Tool-MAD（2026）**强调异构工具与自适应检索，并将 Faithfulness、Answer Relevance 引入最终判断。

**MAD-Fact（2025）**将长文本拆成不同重要性的事实主张，再由 Clerk、Jury、Judge 分工核验。

对本项目的启示：

- 应核验“具体主张”，例如“20 日收益为 8.1%”，而不是笼统评价整篇报告；
- 不需要照搬更多 Agent；当前 Bull/Bear 和 Research Manager 已覆盖辩论与裁决角色；
- 更合适的落地是增加结构化 `ClaimRecord` 和确定性检查。

限制：

- Tool-MAD 的工具异构性不能靠同一 OHLCV 数据换不同包装来伪造；
- LLM Judge 的 Faithfulness 只衡量回答是否被上下文支持，Answer Relevance
  只衡量是否切题；错误、过期或污染来源仍可能得到高分；
- DebateCV 不证明当前未专门训练的 Research Manager 能可靠裁决金融因果、
  估值或预测主张。

### 2.4 可信审计需要执行来源图

**PROV-AGENT（IEEE e-Science 2025）**扩展 W3C PROV，将 Agent、提示、
响应、工具调用和工作流关系纳入统一来源模型。

对本项目的启示：

- 证据应在产生时登记，而不是报告完成后猜测来源；
- CLI 与 `TradingAgentsGraph.save_reports()` 程序化保存路径应共享同一种 trace 结构；
- 数据快照、工具参数、Agent 输出和最终结论应能串成有向关系。

限制：

- 本项目不需要第一阶段完整实现 W3C PROV；
- 应先采用精简 JSONL，再保留转换为 PROV 图的可能。
- Provenance 只证明“记录显示发生了什么”，不证明工具输出真实、来源可信或
  投资结论正确。

### 2.5 不确定性必须经过校准

**DiverseAgentEntropy（Findings of EMNLP 2025）**通过同一问题的知识保持变体和多 Agent 交互估计不确定性，并在高不确定时采用 abstention。

**Belief-Calibrated Consensus Seeking（NeurIPS 2025）**指出，稳定共识不能只靠投票，还应考虑内部信念冲突和协作者选择。

对本项目的启示：

- 第一阶段不要让 Agent 自报一个“信心百分比”；
- 先输出可解释的多维风险画像；
- 只有积累历史预测结果并完成校准后，才可以给出一个统计意义明确的可信概率。

限制：

- 多次调用 LLM 计算 entropy 成本很高，也可能与现有辩论重复；
- 该方法面向可聚类离散答案的知识问答；金融收益预测连续、时变且通常没有
  即时真值，其 entropy 和阈值不能直接移植；
- 当前阶段只设计可选实验，不进入默认运行路径。

### 2.6 金融 Agent 必须做系统级扰动测试

**FINCON（NeurIPS 2024）**在其历史回测和消融实验中报告，经理—分析师层级
及 CVaR 触发的自我批评/概念化信念更新改善了部分实验指标；这不能推出实盘
收益、稳健 Alpha 或本项目可信度必然提高。

**TradeTrap（2025，arXiv 预印本）**在其有限股票、有限时间段和两类 Agent
设置下观察到局部扰动传播和风险放大，支持开展系统级故障注入，但不代表所有
市场与 Agent 都会产生相同结果。

对本项目的启示：

- 可信度验收不能只看正常运行；
- 必须测试过期数据、缺失工具、冲突来源、错误数字、结构化输出失败和记忆污染；
- 评估指标应包括错误传播和检测能力，而不只看收益。

## 3. 当前系统已有能力与真实缺口

### 3.1 已有能力（应复用，不重复开发）

1. **as-of 数据路径**  
   OHLCV 和新闻调用均携带分析日期，Market 还具有 `get_verified_market_snapshot`。

2. **部分节点已有结构化输出**  
   Sentiment、Research Manager、Trader、Portfolio Manager 已有 Pydantic
   schema；Market、News、Fundamentals 仍以工具循环后的自由文本报告为主。
   结构化调用失败时会回退自由文本。

3. **Rating 统一解析**  
   `parse_rating` 支持五档评级及中英文回退。

4. **分析师 ToolNode**  
   Market、News、Fundamentals 具备 ReAct 工具循环；Sentiment 使用确定性新闻预取和词典打分。

5. **Bull/Bear 非重叠工具**  
   相对基准和回撤工具每次完整 run 各最多执行一次，并通过 `used_tools` 管理调用预算。

6. **确定性报告附录**  
   保存报告时生成多周期、Alpha、回撤和价量图，不经过 LLM。

7. **基础日志与 checkpoint**  
   CLI 会写 `message_tool.log`；图支持 SQLite checkpoint；报告树保存各阶段文本。

8. **需要单独清理的历史量化语义**  
   默认图已经不消费 BHD，但 `QuantSignal` 仍保留 `suggested_weight` /
   `confidence`，统计附录仍展示 `q_score`。可信度实现禁止读取这些仓位字段；
   应另立清理任务删除仓位语义，或将其严格降为只供展示的统计特征。

### 3.2 缺口

1. CLI 日志和 `TradingAgentsGraph` 程序化调用方没有统一的工具来源记录；
2. Tool 输出没有稳定的 observation/artifact id 与字段 selector，下游无法精确引用；
3. 长报告中的数字、日期和因果主张没有结构化登记；
4. structured-output 回退发生时只有日志 warning，最终报告看不到该降级；
5. Research Manager 没有显式列出接受、拒绝和未决主张；
6. Trader、PM 的方向变化没有机器可读解释；
7. 报告附录虽然确定性生成，但没有自动核对正文数字；
8. 记忆条目缺少“来自哪次证据/哪次决策”的来源关系；
9. 没有统一审计状态和故障压力测试；
10. 公共层已有 `TraceEvent`、`DecisionEnvelope.trace/diagnostics`，新设计必须
    复用或映射这些契约，不能再制造一套公共真相。

## 4. 设计总览：五层能力，而非八条独立流水线

```text
数据 / Tool
  │
  ├─ ① 采集层：Observation / Artifact / Trace
  │
Analyst 输出
  │
  ├─ ② 声明层：Typed Claim / Decision Snapshot
  │
Bull / Bear
  │
  ├─ ③ 统一规则引擎：
  │     numeric / temporal / source / debate / handoff rule packs
  │
Research Manager → Trader → Risk → Portfolio Manager
  │
最终状态
  │
  └─ ④ 呈现层：Audit Profile + Audit Bundle

⑤ 验证工程（不在生产运行链重复执行）：故障注入、回归、消融与校准
```

运行时只有一个规范事件流、一个声明集合和一个规则引擎。Debate 与 Handoff
只是规则包；Profile 与 Audit Bundle 只是同一审计结果的人读/机读投影；
Stress Evaluation 是测试工程。这样可避免多个模块重复扫描长文本、重复产生
Finding 或维护不一致状态。

## 5. 第一层：Evidence / Trace 采集

### 5.1 目标

在工具结果产生的瞬间登记来源。它只记录来源，不判断投资结论。

### 5.2 数据结构

“本次调用”和“可复用载荷”必须分开，不能仅存一个 hash + 摘要：

```python
class EvidenceObservation:
    run_id: str
    event_id: str
    tool_call_id: str
    producer_node: str
    tool_name: str
    provider: str
    arguments_redacted: dict
    symbol: str
    decision_cutoff_at: datetime
    started_at: datetime
    ended_at: datetime
    attempt: int
    cache_status: str
    artifact_id: str | None
    parent_event_id: str | None
    status: "SUCCESS" | "EMPTY" | "ERROR" | "STALE"
    error: str | None

class EvidenceArtifact:
    artifact_id: str
    media_type: str
    schema_name: str
    schema_version: str
    canonical_hash: str
    payload_ref: str
    provider_identity: str
    occurred_at: datetime | None
    published_at: datetime | None
    observed_at: datetime

class ArtifactFieldRef:
    observation_event_id: str
    artifact_id: str
    selector: str               # JSON Pointer 或有版本的字段选择器
    schema_name: str
    schema_version: str
```

每次调用都写 `EvidenceObservation`；只有内容完全相同的 Artifact 可以去重。
Claim 引用 `observation_event_id + artifact_id + selector`，既绑定“本次运行看到了
什么”，也绑定“载荷中的哪个字段”。
`canonical_hash` 只能证明捕获后的完整性，不能证明上游数据真实或来源独立。

### 5.3 插入点

- 在 `AgentState` 增加 `run_id` 与审计事件引用；
- 提供 `EvidenceSink(run_id)` 和统一 `invoke_evidenced()`；
- ToolNode 通过包装每个 BaseTool 接入；
- Sentiment 强制预取、Bull/Bear 工具循环、报告附录直接函数调用显式走同一入口；
- `event_id` 按 run/node/tool_call/attempt 幂等生成，checkpoint 恢复使用 upsert。

不能只包装 ToolNode，因为当前实际有 ToolNode、Sentiment 直接调用、Bull/Bear
自定义 tool loop、附录直接 Python 调用四种路径。

### 5.4 输出

每次运行生成：

```text
audit/event_trace.jsonl
audit/artifacts/
```

每次观测都保留；相同载荷只复用 Artifact，不能丢失“本次运行确实观测过”的事实。

目标实现以公共 `TraceEvent` 作为唯一事件 envelope：

- `timestamp/source/event_type/summary` 使用现有公共字段；
- `payload` 按版本写入 run_id、event_id、tool_call_id、artifact_id 等；
- `event_trace.jsonl` 是 `TraceEvent` 的规范序列化，不再维护另一套事件模型；
- Observation 是 `event_type=EVIDENCE_OBSERVATION` 的 typed payload view；
- claims/findings/profile 都是由该事件流和 Artifact 派生的索引。

### 5.5 有效性如何体现

- 随机抽取报告数字，可以反查到调用事件、载荷字段、参数和截止时间；
- CLI 与调用 `TradingAgentsGraph.save_reports()` 的程序化路径产生相同格式；
- 空数据和工具异常不再被报告悄悄吞掉。

### 5.6 不做什么

- 不判断新闻是否真实；
- 不给证据打“看涨/看跌”分；
- 不重复保存完整大文本，可保存 hash、摘要和缓存引用。

## 6. 第二层：Claim / Decision 声明契约

### 6.1 目标

将长报告中真正影响决策的内容变成可核验对象。

### 6.2 主张类型

```python
class VerificationStatus(str, Enum):
    NOT_EVALUATED = "NOT_EVALUATED"
    SUPPORTED = "SUPPORTED"
    CONTRADICTED = "CONTRADICTED"
    UNSOURCED = "UNSOURCED"
    NOT_DETERMINISTICALLY_VERIFIABLE = "NOT_DETERMINISTICALLY_VERIFIABLE"

class ClaimRecord:
    claim_id: str
    identity_key: str  # 由规范化 subject/predicate/value/unit/window/as_of 生成
    run_id: str
    agent: str
    stage: str
    text: str
    claim_type: "NUMERIC" | "DATE" | "FACT" | "CAUSAL" | "FORECAST" | "OPINION"
    importance: "CRITICAL" | "MAJOR" | "MINOR"
    subject: str | None
    predicate: str | None
    value: float | str | None
    unit: str | None
    period_start: datetime | None
    period_end: datetime | None
    as_of: datetime | None
    evidence_refs: list[ArtifactFieldRef]  # artifact_id + JSON Pointer/字段 selector
    tolerance: float | None
    polarity: str | None
    inherited_claim_ids: list[str]
    capture_status: "STRUCTURED" | "FALLBACK_CANDIDATE"
    verification_status: VerificationStatus
```

分类意义：

- `NUMERIC` / `DATE` 必须填结构化字段，才能由代码强核验；
- `FACT` 可检查是否有可定位来源；无来源记为 `UNSOURCED`，有来源但无法由
  代码判断真值时记为 `NOT_DETERMINISTICALLY_VERIFIABLE`；只有字段级核验
  通过/冲突时才记为 `SUPPORTED/CONTRADICTED`；
- `CAUSAL` / `FORECAST` 不能被代码证明，只能要求证据与不确定性说明；
- `OPINION` 不应伪装成事实。

### 6.3 生成方式

第一阶段不增加“Claim Extractor Agent”。先让已有结构化输出的 Sentiment、
Research Manager、Trader、PM 返回 sidecar：

- `key_claims`；
- `evidence_refs`；
- `uncertainties`。

Market、News、Fundamentals 先保留正文，由工具调用产生 typed evidence；下一阶段
再逐节点设计结构化 final-report schema。自由文本规则只能抽取候选数字/日期，
状态必须标记为 `FALLBACK_CANDIDATE`，不能宣称覆盖了全部关键主张。

### 6.4 审计覆盖边界

“关键主张全集”定义为：

1. 所有结构化 sidecar 中 importance=CRITICAL/MAJOR 的 Claim；
2. 所有 Numeric/Date Claim；
3. Recommendation、Action、Rating、Price Target、Entry、Stop Loss 等决策字段；
4. 被 RM、Trader 或 PM 实际引用的上游 Claim。

如果任一被下游消费的节点仍是自由文本且没有完整 sidecar，或发生 structured
fallback，则 `audit_scope=PARTIAL`。在 `audit_scope=PARTIAL` 时，最高只能得到
`AUDIT_DEGRADED`，不能得到 `AUDIT_PASSED`。规则抽取到的候选主张可以帮助发现
问题，但不能证明未抽取主张不存在。

### 6.4 有效性如何体现

- 已进入 ClaimRecord 的关键主张证据覆盖率可以直接计算；
- 读者能区分“事实”“预测”“观点”；
- 不需要另一次 LLM 调用，也不会新增与现有 Agent 重复的角色。

## 7. 第三层：统一规则引擎

### 7.1 目标

只核验代码能够客观判断的内容，不假装理解所有金融因果关系。

### 7.2 检查规则

#### 数字一致性

- 按 Claim 的 `artifact_id + selector` 取得证据字段，不做全文数字搜索；
- 允许明确容差，例如收益率绝对误差不超过 0.1 个百分点；
- 单位是否一致：比例、百分比、美元、股数；
- 字段含义不能改写，例如 126 根窗口最大回撤不能写成“20 日最大回撤”。

#### 日期安全

- 使用 timezone-aware 的 `occurred_at / published_at / observed_at /
  decision_cutoff_at`；
- 按资产交易日历和数据源 latency policy 判断信息在决策时点是否可用；
- 报告中的未来日期只能作为计划/预测出现；
- 新闻发布日期晚于 decision cutoff 才记为 `TEMPORAL_VIOLATION`。

#### 来源健康

- 引用的 `artifact_id` 和字段 selector 必须存在；
- Tool 状态为 EMPTY/ERROR 时不得作为肯定证据；
- 数据 provider 与报告披露一致。

#### 结构化输出完整性

- Rating、Action、Recommendation 必须属于枚举；
- structured fallback 必须显式写入运行状态；
- 缺少关键字段不得被当作完全可信。

### 7.3 输出

```python
class VerificationFinding:
    finding_id: str
    rule_id: str
    rule_version: str
    status: str
    code: str
    severity: "INFO" | "WARNING" | "CRITICAL"
    stage: str
    claim_id: str | None
    artifact_id: str | None
    expected: object | None
    actual: object | None
    location: str | None
    first_seen_event_id: str
    message: str
```

### 7.4 与 Evidence Ledger 的区别

- Ledger 证明“记录显示某次调用产生/取得了这个载荷”；
- Verifier 判断“报告主张是否正确引用了这个数据”。

因此两者不是重复模块；但所有规则包只由同一个 Verifier 运行并产生一次 Finding。

## 8. 规则包 A：Truth-Seeking Debate（求真辩论）

### 8.1 目标

把 Bull/Bear 从“为了赢而说服”改为“寻找当前投资论证中最脆弱的主张”。

### 8.2 保留现有结构

不增加新 Debater，不增加多数投票。继续使用：

- Bull；
- Bear；
- Research Manager。

### 8.3 新规则

每轮发言要求包含：

1. 本轮针对的 `claim_id`；
2. 支持、反驳或限定该主张；
3. 新增 artifact field ref，或明确说明只是在重新解释已有证据；
4. 对手论证中最具体的错误；
5. 本方仍无法解决的不确定性。

每轮同时输出：

```python
class DebateTurnRecord:
    target_claim_id: str
    stance: str
    added_evidence_refs: list[ArtifactFieldRef]
    changed_claim_status: list[ClaimStatusChange]
    correction_codes: list[CorrectionCode]

class ClaimStatusChange:
    claim_id: str
    old_status: VerificationStatus
    new_status: VerificationStatus

class CorrectionCode(str, Enum):
    WRONG_VALUE = "WRONG_VALUE"
    WRONG_UNIT = "WRONG_UNIT"
    WRONG_WINDOW = "WRONG_WINDOW"
    STALE_EVIDENCE = "STALE_EVIDENCE"
    UNSUPPORTED_CLAIM = "UNSUPPORTED_CLAIM"
    SEMANTIC_MISLABEL = "SEMANTIC_MISLABEL"
```

只有结构化集合差分表明没有新 claim、evidence、状态变化或 correction code 时，
才记录 `NO_NEW_CLAIM_OR_EVIDENCE` 并允许提前结束。LLM 自述“发现了新错误”
不能直接触发路由变化。实现需同时修改 debate state、Bull/Bear 节点和
conditional logic。

判定单位是一个完整 Bull+Bear cycle，而不是单次发言。去重身份分别为：

- Claim：稳定的 `identity_key`（不是每轮可重新生成的 claim_id）；
- Evidence：`(observation_event_id, artifact_id, selector)`；
- Correction：`(target_claim_id, correction_code)`；
- 状态变化：`(claim_id, old_status, new_status)`。

默认 `novelty_patience=1`：连续一个完整 cycle 没有新增上述身份才提前停止；
该值写入配置和 manifest。只有一方完成发言、调用失败或 checkpoint 中断的
不完整 cycle 不参与停止判定。

### 8.4 工具边界

继续只保留：

- `compare_to_benchmark_for_ticker`；
- `get_drawdown_risk_stats_for_ticker`。

每次完整 run 各最多执行一次。Market 与 Sentiment 已经提供的工具不在辩论层重复调用。

### 8.5 Research Manager 的裁决职责

扩展 `ResearchPlan`：

```python
accepted_claim_ids: list[str]
rejected_claim_ids: list[str]
unresolved_claim_ids: list[str]
decision_change_reason: str | None
```

Research Manager 不按票数决定胜负，而按：

- 证据是否存在；
- 数字是否通过核验；
- 证据是否新鲜；
- 反方是否成功指出具体错误；
- 是否仍有关键未决风险。

### 8.6 有效性如何体现

- 每轮新增主张/证据数量（debate novelty）；
- 重复发言率；
- 被对手纠正的错误数量；
- Research Manager 引用的有效 evidence 数；
- 辩论轮数增加时， unsupported claim rate 不应上升。

## 9. 规则包 B：Handoff Consistency（节点交接一致性）

### 9.1 目标

检测 Research Manager → Trader → Risk → PM 中没有解释的方向变化。

### 9.2 检查内容

1. Recommendation、Action、Rating 是否可解析；
2. 决策节点是否对上游关键结论的改变给出显式理由；
3. PM 是否回应 Risk 三方提出的关键风险；
4. Price Target、Entry、Stop Loss 的数值关系；
5. PM 是否使用了已被 Research Manager 拒绝的 claim；
6. PM 是否忽略 critical unresolved claim。

### 9.3 当前阶段不做方向硬映射

Overweight/Underweight 是相对基准或目标暴露，Buy/Sell 是交易动作，Hold 是
不动作；没有当前仓位和目标仓位时，不能把它们可靠映射到同一
`bullish/neutral/bearish` 轴。

当前只检查字段可解析和显式理由。PM 可以通过引用既有 evidence_ref、Risk
阶段新增的 `constraint_id` 或明确的 portfolio/memory constraint 解释变化，
不得要求没有 Tool 的 PM 产生新证据。

进入 Broker 集成后，才使用 `(current_weight, target_weight,
benchmark_weight)` 判断动作一致性。集成前只产生 `REVIEW_NEEDED`，不自动
产生“无解释方向冲突”严重告警。

### 9.4 不做什么

- 不强制所有节点意见一致；
- 不把分歧本身当作低可信；
- 不修改 PM Rating。

## 10. 第四层：Audit Profile 与审计呈现

### 10.1 为什么不直接给一个总分

在没有历史校准之前，“可信度 87%”是伪精确。第一阶段输出多维画像：

```python
class AuditProfile:
    run_id: str
    policy_version: str
    generated_at: datetime
    audit_scope: "FULL" | "PARTIAL"
    audit_scope_reasons: list[str]
    status: "AUDIT_PASSED" | "AUDIT_DEGRADED" | "AUDIT_FAILED"
    evidence_coverage: float | None
    numeric_verification_rate: float | None
    temporal_integrity: float | None
    handoff_explanation_coverage: float | None
    tool_success_rate: float | None
    structured_output_integrity: float | None
    debate_novelty_rate: float | None
    metric_counts: dict
    critical_findings: list[str]
    warnings: list[str]
```

每个比例必须同时保存 `eligible_count / excluded_count / unknown_count`。
可选 Analyst、非美股/加密资产不适用的 Fundamentals、合法空新闻等不能进入
错误分母；EMPTY 按工具 policy 区分 `EXPECTED_EMPTY` 与
`UNEXPECTED_EMPTY`。每个字段可单独诊断，不随意加权成总概率。

指标公式固定为：

- `evidence_coverage = 有至少一个有效 ArtifactFieldRef 的 eligible 关键主张数 /
  eligible 关键主张数`；
- `numeric_verification_rate = verification_status=SUPPORTED 的 eligible
  Numeric/Date Claim 数 / 已完成确定性核验的 eligible Numeric/Date Claim 数`；
- `temporal_integrity = 无 temporal violation 的 eligible Observation 数 /
  已完成时间核验的 eligible Observation 数`；
- `tool_success_rate = SUCCESS 或 policy 允许的 EXPECTED_EMPTY 调用数 /
  eligible 工具 attempt 数`；
- `structured_output_integrity = mode=STRUCTURED 的必需节点数 /
  本次选择下的必需结构化节点数`；
- `debate_novelty_rate = 至少新增一个去重身份的完整 cycle 数 / 已完成 cycle 数`；
- `handoff_explanation_coverage = alignment=MODIFIED 且具有 reason/constraint ref
  的 Decision Snapshot 数 / alignment=MODIFIED 的 Decision Snapshot 数`。

Decision Snapshot 必须显式记录
`alignment=ADOPTED|MODIFIED|NOT_COMPARABLE`、`upstream_decision_ref` 和
`change_reason_refs`。该指标只衡量“改变是否有解释”，不判断解释是否正确，也
不把五档 Rating 与三档 Action 强行映射。分母为 0 时指标为 `N/A`，不能记为
100%。

### 10.2 状态规则

#### AUDIT_PASSED

- 无 CRITICAL；
- 已进入声明契约的关键主张均有可定位来源；
- 日期约束通过；
- 关键结构化输出未降级；
- 适用的审计规则均通过。

这只表示审计完整性检查通过，不表示来源内容真实、因果关系成立、预测正确或
投资建议可靠。

#### AUDIT_DEGRADED

- 存在工具空结果、自由文本 fallback、部分主张无证据等；
- 报告可阅读，但必须显示警告。

合法 fallback 只降低 `structured_output_integrity`，不会自动升级为失败。

#### AUDIT_FAILED

- 未来数据泄漏；
- 关键数字与证据显著冲突；
- 关键 Evidence 丢失；
- 审计链损坏。

状态只决定“报告可信标签”，不改变投资方向。

### 10.3 可选 abstention

默认行为：

- 仍保存 PM 原始 Rating；
- 报告顶部显示 `AUDIT_FAILED — 审计完整性检查未通过`。

PM 是 final Rating 的唯一权威；Broker Policy 是独立的执行安全权威。
AuditProfile 只能进入 `DecisionEnvelope.diagnostics`，警告写入
`DecisionEnvelope.diagnostics["warnings"]`，不得自行生成
TradeIntent、target_weight 或改写 Rating。若未来 Broker Policy 因审计失败
拒绝执行，必须保留 PM 原始 Rating，并记录独立 `policy_decision` 和
`reason_code`，不得把它称为 PM 结论。

## 12. 第五层：Stress Evaluation（验证工程）

### 12.1 测试场景

#### 数据层

- OHLCV 缺失；
- 新闻为空；
- benchmark 不可用；
- 返回分析日之后的数据；
- provider 返回格式变化。

#### 工具层

- Tool 抛异常；
- Tool 超时；
- 同一工具重复调用；
- Tool payload 被篡改；
- 两个来源数值冲突。

#### Agent 层

- Market 报告编造百分比；
- Bull/Bear 把 126 根窗口误称为 20 日；
- Research Manager 忽略关键反例；
- Trader 与 Research 方向相反但无解释；
- PM structured output 失败并回退；
- 多轮辩论重复同一观点；
- 记忆中注入未来信息。

#### 系统层

- checkpoint 恢复后 audit event/artifact 引用丢失；
- CLI 保存路径与 `TradingAgentsGraph.save_reports()` 程序化保存路径不一致；
- 同 ticker/date 重跑不能复现确定性附录；
- 报告文件被修改后 manifest hash 不匹配。

### 12.2 评估指标

1. **Critical Detection Recall**：注入的严重错误被检测比例；
2. **False Alarm Rate**：合法分歧被错误报警比例；
3. **Evidence Coverage**：适用的关键主张具有有效 artifact field ref 的比例；
4. **Numeric Verification Rate**：数值主张通过确定核验的比例；
5. **Temporal Violation Rate**：未来数据违规比例；
6. **Handoff Explanation Coverage**：声明为 MODIFIED 的交接中具有
   reason/constraint ref 的比例；
7. **Debate Novelty**：每轮新增有效 claim/evidence 数；
8. **Propagation Depth**：错误在被发现前传播了多少节点；
9. **Audit Reproducibility**：相同输入下确定性审计产物一致率。

收益率、Sharpe 和回撤仍可用于交易效果评估，但不能替代上述可信度指标。
Stress fixtures 从第一阶段起随每项能力同步建设，不是最后才运行的独立生产组件。

验收：

- 针对明确二元目标报告 Brier Score、ECE、coverage-risk 曲线；
- 阈值来自验证集，不由开发者拍脑袋设定；
- 不把“报告事实正确”和“股票后来上涨”混成一个可信标签。

---

## 15. 明确删除或拒绝的重复设计

1. **不增加第二套 Market/Quant 分析器**  
   所有市场统计复用现有 Tool 和报告附录。

2. **不增加独立 LLM Judge 来重写 PM 结论**  
   Research Manager 已是辩论裁决者，PM 已是最终决策者。

3. **不让 Bull/Bear 重调 Analyst 已有工具**  
   只保留相对基准和回撤两个非重叠工具。

4. **不把 provenance 与 verification 合并为重复流水线**  
   一个记录来源，一个核对主张。

5. **不同时维护“可信度总分”和“风险分”**  
   第一阶段只保留多维 AuditProfile。

6. **不把 Agent 共识当作事实验证**  
   同模型共识可能是相关错误。

7. **不保存或宣称验证隐藏思维链**  
   只记录可观察输入、工具、结构化主张、输出和关系。

8. **不恢复 BHD**  
   可信度状态不改变买卖方向，也不计算仓位。

## 16. 多轮审查清单

### 第一轮：职责审查

- 每个组件是否只有一个核心职责？
- 是否出现两处重复计算同一市场指标？
- 是否新增了与 RM/PM 重复的裁判？

### 第二轮：失败模式审查

- Tool 为空或报错时会不会被当成成功？
- structured fallback 是否可见？
- 未来数据是否能进入报告？
- 合法分歧是否会被误判为错误？

### 第三轮：可测性审查

- 每个组件是否有输入、输出和验收指标？
- 是否可以通过 fixture 注入错误？
- 是否能比较增强前后差异？

### 第四轮：产品语义审查

- PM 是否仍是最终 Rating 权威？
- Audit Profile 是否只做披露和审计？
- Broker Policy 的执行安全权是否与 PM 的评级权清楚分离？

## 18. 最终建议

最值得优先开发的不是“额外 Agent”，而是：

1. Evidence Ledger；
2. structured fallback 可见化；
3. 确定性数字/日期核验；
4. RM→Trader→PM 方向变化解释；
5. audit bundle；
6. 压力测试。

这六项成本相对可控、功能不重叠，而且能立刻在报告中体现价值。Claim-level 辩论和不确定性校准应在审计基础稳定后再做，否则会在缺少可靠来源数据时堆叠更多 LLM 输出。
