## 0. 决策摘要

| 项 | 结论 |
|----|------|
| BHD / 硬编码融合层 | **删除**（provider / fusion / boundaries / bridge 闸门、第 VI 节、`executable_decision`） |
| 已实现的分析师工具 | **保留并扩展**（多周期 OHLCV、量能异常、缺口、词典情绪等） |
| 与 tools 同构的「第二套 Quant 决策引擎」 | **不再维护**；特征函数仅服务 tools，不产出执行仓位 |
| 最终决策权威 | **多智能体 + Portfolio Manager**；信号解析回退为 `final_trade_decision` |
| 幻觉与可控性 | **不靠再算一遍指标**；用工具证据绑定、结构化 Rating、报告校验、可选裁判节点 |

## 2. 架构（清理后）

```text
Analysts (Tools → 报告)
    → Bull / Bear → Research Manager
    → Trader → Risk 三人 → Portfolio Manager
    → 解析 Rating / 写报告树 / 记忆
```

- **不再**在 PM 后插入融合仓位闸门。  
- Tools 只服务「看得清市场」；LLM 负责辩论与评级。  
- 若未来需要「纸面仓位」，另开 **极薄 Policy**（纯 Rating→权重映射），**禁止**再引入与 tools 平行的全套特征引擎——本方案默认不做，除非产品明确要接 Broker。

### 3.3 工作流里工具怎么被选中？（按需，不是全调）

四个 Analyst 使用图内 ToolNode；Bull/Bear 通过节点内轻量工具循环共享两个辩论工具。
从 Research Manager 开始的下游节点不再调用 Tool。

对每个分析师节点，流程是 **ReAct 循环**：

1. LLM 看到「可用工具列表」+ 系统词（哪些建议/必须用）。  
2. 模型自己决定要不要 `tool_calls`、调用哪几个、参数是什么。  
3. 若有 `tool_calls` → 图走到对应 `tools_*` 节点执行 → 结果回写 messages → **再回到同一分析师** 继续想。  
4. 若本轮 **没有** tool_calls → 清消息 → 进入下一个 Agent。

因此：

| 问题 | 答案 |
|------|------|
| 每个工具都会调用吗？ | **不会。** 以 LLM 当次决策为准；提示词可「强烈建议」，但不能 100% 保证。 |
| Market 的 6 个工具？ | **按需**；提示词要求先 `get_stock_data`、至少用一次 C 的三件套、写报告前用 `get_verified_market_snapshot`，但仍可能漏调或少调。 |
| Sentiment 的词典？ | **每次都会跑**：代码里先 `get_news` + `score_documents`，再喂给 LLM——这不是模型自选 Tool。 |
| 下游 PM 等？ | **零工具调用**，只读上游报告。 |

## 4. 后续扩展（仍在 C 工作范围）

按优先级，不恢复 BHD：

### P0 — 报告可信度

- 详细架构、研究依据、组件边界和验收标准见 **`多智能体可信度设计方案.md`**  
- PM / Trader 强制结构化字段：`**Rating**` / `Action` 可解析  
- **已实现基础版**：统一 Trace/Artifact、结构化降级、Typed Claims、确定性
  Verifier、AuditProfile、完整 `audit/` 审计包、求真辩论记录与交接解释；
  全部只做披露与诊断，不改 PM Rating  
- 保存报告后新增 `audit/event_trace.jsonl`、`artifacts/`、`claims.jsonl`、
  `findings.jsonl`、`profile.json`、`manifest.json`、`credibility.md`  

### P1 — 报告内嵌时序图（你点名的需求）

- **已实现**：`report_appendix.py` 在保存报告时写入 **第 VI 节附录**（确定性统计表 + 价量图）  
- 目录：`reports/.../appendix/statistics.md`、`appendix/price_volume.png`  
- 依赖：`matplotlib`（已加入 `pyproject.toml`）  
- 扩展：标注缺口日、量能异常日、多标的对比小 multiples  

### P2 — 工具扩展（避免与「决策硬编码」混淆）

优先补 **Analyst 证据**，例如：

- 波动率历史分位（高波日提示）  
- 距下次财报天数（事件风险）  
- 情绪分时序（若有多日新闻缓存）  

全部以 **Tool 返回 JSON/表** 形式给 Agent，**不**再做独立融合仓位模块。

### P3 — 模拟盘 / 评估（与 Brokers 协作时）

- 用 PM Rating 映射到粗粒度动作即可起步  
- 评估「有工具 vs 无工具」报告质量与稳定性，而不是「有 BHD vs 无 BHD」

### P4 — 文档与协作

- 保持 `数据源与分析师清单.md` 与本文同步  
- `DecisionProvider` 等 **A/B/C 公共契约**可保留为空协议，供日后 Broker 接入；**C 不再提供 HybridDecisionProvider 实现**
